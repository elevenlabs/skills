# Real-Time Clinical Voice Alerts

Speak urgent, text-driven clinical events (missed screenings, abnormal vitals, critical labs, overdue tasks) to clinicians with low enough latency that the alert is still actionable when it arrives.

## When to Use This Pattern

Use a one-shot TTS alert, not a conversational agent, when:

- The system is telling a clinician something, not conversing with them.
- The text is generated by your backend (a rules engine, a scheduled job, a lab result webhook), not typed live by a user.
- Latency matters more than expressiveness — a flat, clear, fast "Missed PHQ-9 screening detected for patient in Room 4B" beats a warmer-sounding alert that arrives two seconds later.

## Voice Selection for Clinical Contexts

Pick a stable, clearly-articulated voice and lock down its settings so the alert never sounds ambiguous or overly emotive. For alerts, prioritize **stability** over expressiveness:

```python
from elevenlabs import VoiceSettings

CLINICAL_ALERT_VOICE_SETTINGS = VoiceSettings(
    stability=0.75,          # high stability: consistent, predictable delivery
    similarity_boost=0.8,
    style=0.0,                # no style exaggeration for alerts
    speed=1.0,
    use_speaker_boost=True,
)
```

Good starting voices for clinical alerts (clear, professional, low ambiguity):

- `onwK4e9ZLuTAKqWW03F9` - Daniel (male, authoritative)
- `EXAVITQu4vr4xnSDxMaL` - Sarah (female, soft but clear)
- `JBFqnCBsd6RMkjVDRZzb` - George (male, narrative)

List available voices programmatically and let a clinical admin pick one at configuration time, rather than hardcoding a single voice for every deployment:

```python
from elevenlabs import ElevenLabs

client = ElevenLabs()

voices = client.voices.get_all()
for voice in voices.voices:
    print(f"{voice.voice_id}: {voice.name}")
```

## Latency Optimization (Target: Under 2 Seconds)

Three levers control end-to-end latency for a spoken alert:

1. **Model choice.** Use `eleven_flash_v2_5` (multilingual, ~75ms model latency) or `eleven_flash_v2` (English-only, ~75ms). Do not use `eleven_multilingual_v2` or `eleven_v3` for real-time alerts — they trade latency for quality you don't need here.
2. **`optimize_streaming_latency`.** Both `convert` and `stream` accept an integer from 0-4. For clinical alerts, use `3` (max latency optimizations, default text normalizer) or `4` if you've already normalized the text yourself and want the absolute lowest latency:

```python
audio = client.text_to_speech.convert(
    voice_id="onwK4e9ZLuTAKqWW03F9",
    text="Missed PHQ-9 screening detected for patient in Room 4B.",
    model_id="eleven_flash_v2_5",
    optimize_streaming_latency=3,
    output_format="pcm_24000",   # low-latency raw format; decode/play directly
    voice_settings=CLINICAL_ALERT_VOICE_SETTINGS,
)
```

3. **Streaming instead of waiting for the full file.** `stream()` yields audio chunks as they're generated, so playback can start before the full alert has finished synthesizing:

```python
audio_stream = client.text_to_speech.stream(
    voice_id="onwK4e9ZLuTAKqWW03F9",
    text="Critical potassium value resulted for patient in Room 4B.",
    model_id="eleven_flash_v2_5",
    optimize_streaming_latency=3,
    output_format="pcm_24000",
)

for chunk in audio_stream:
    play_audio_chunk(chunk)  # push to your audio output as it arrives
```

Measure and log actual latency in production (see [evaluation.md](evaluation.md) for a benchmarking pattern) — the ~75ms figure is model generation time only and does not include network round-trip or your own audio playback pipeline.

## Text Normalization for Alerts

Clinical alert text often contains numbers, abbreviations, and room/unit identifiers that must be spoken unambiguously. Use `apply_text_normalization="on"` unless you've pre-formatted the text yourself, in which case `"off"` avoids the (small) added latency of the normalizer:

```python
audio = client.text_to_speech.convert(
    voice_id="onwK4e9ZLuTAKqWW03F9",
    text="Potassium 6.2 milliequivalents per liter, critical high, Room 4B.",
    model_id="eleven_flash_v2_5",
    apply_text_normalization="off",   # already spelled out for clarity
    optimize_streaming_latency=3,
)
```

If you pass raw values instead ("K+ 6.2 mEq/L"), use `"on"` and let the model normalize it, but expect slightly higher latency.

## FastAPI Endpoint: Clinical Alert to Audio

A minimal endpoint that takes de-identified alert text and returns audio. Note what it does **not** do: it does not accept patient names, MRNs, or DOB in the request — see [SKILL.md](../SKILL.md#phi--hipaa-considerations) for why.

```python
import os
from fastapi import FastAPI, HTTPException
from fastapi.responses import StreamingResponse
from pydantic import BaseModel, Field
from elevenlabs import ElevenLabs, VoiceSettings

app = FastAPI()
client = ElevenLabs(api_key=os.environ["ELEVENLABS_API_KEY"])

ALERT_VOICE_ID = "onwK4e9ZLuTAKqWW03F9"  # Daniel - clear, authoritative
ALERT_MODEL_ID = "eleven_flash_v2_5"     # ~75ms latency

ALERT_VOICE_SETTINGS = VoiceSettings(
    stability=0.75,
    similarity_boost=0.8,
    style=0.0,
    speed=1.0,
    use_speaker_boost=True,
)

MAX_ALERT_CHARS = 300  # keep alerts short; this is a notification, not a note


class ClinicalAlertRequest(BaseModel):
    # De-identified alert text only. Do not include patient name, MRN, or DOB.
    # e.g. "Missed PHQ-9 screening detected for patient in Room 4B."
    alert_text: str = Field(..., max_length=MAX_ALERT_CHARS)


@app.post("/clinical-alerts/speak")
def speak_clinical_alert(request: ClinicalAlertRequest):
    """Convert a de-identified clinical alert to audio and stream it back."""
    if not request.alert_text.strip():
        raise HTTPException(status_code=400, detail="alert_text must not be empty")

    try:
        audio_stream = client.text_to_speech.stream(
            voice_id=ALERT_VOICE_ID,
            text=request.alert_text,
            model_id=ALERT_MODEL_ID,
            optimize_streaming_latency=3,
            output_format="mp3_44100_128",
            voice_settings=ALERT_VOICE_SETTINGS,
            apply_text_normalization="on",
            enable_logging=False,  # zero-retention mode; see SKILL.md PHI section
        )
    except Exception as exc:
        raise HTTPException(status_code=502, detail=f"TTS generation failed: {exc}") from exc

    return StreamingResponse(audio_stream, media_type="audio/mpeg")
```

Run it with `uvicorn main:app --reload` and test with:

```bash
curl -X POST "http://localhost:8000/clinical-alerts/speak" \
  -H "Content-Type: application/json" \
  -d '{"alert_text": "Missed PHQ-9 screening detected for patient in Room 4B."}' \
  --output alert.mp3
```

For a system that must speak alerts continuously (e.g., a nurse-station monitor), keep a single long-lived process holding the `ElevenLabs` client rather than re-instantiating it per alert, and queue alerts so overlapping triggers don't talk over each other.

## Error Handling

```python
try:
    audio = client.text_to_speech.convert(
        voice_id=ALERT_VOICE_ID,
        text=alert_text,
        model_id="eleven_flash_v2_5",
    )
except Exception as e:
    # Fall back to a visual-only alert (dashboard banner, page) rather than
    # silently dropping a clinical notification.
    log_alert_fallback(alert_text, error=str(e))
```

Common errors: **401** (invalid API key), **422** (invalid `voice_id`/`model_id`), **429** (rate limit — have a visual-alert fallback path so a rate limit never means a missed clinical notification).
